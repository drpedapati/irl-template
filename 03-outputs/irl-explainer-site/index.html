IRL: Idempotent Research Loop

I got tired of research workflows that vanish into chat history. IRL is a simple pattern: write your intent in a plan file, let an AI execute it, and let git diffs be the audit trail.

What IRL Is (And What It Isn’t)

IRL is not “a better chat.” It’s a workflow that makes AI-assisted work reproducible.

IRL is

A document-centric loop where the plan file is your control surface. You edit it to change behavior, not the AI’s memory.

Outputs land in folders. Changes show up as diffs. You can review, revert, or iterate.

IRL isn’t

A promise that the AI will “remember forever.” It’s the opposite: assume it won’t, then write down what matters in files you can version.

The point is control and traceability, not vibes.

Why the word “idempotent”?

In IRL, you want reruns to be safe. If you re-execute a step, it should not duplicate outputs, corrupt state, or create mystery changes. Determinism beats heroics.

The IRL Pipeline

The “loop” is small on purpose: intent lives in a plan; evidence lives in files; diffs tell you what changed.

Fig 1. The IRL loop: you edit a plan (intent), the AI runs it (execution), artifacts land in outputs (evidence), then you review diffs and commit (audit trail). Repeat until the plan is done.

What makes this different from “just write notes”?

The plan is executable intent. It’s not a diary. A good IRL plan is concrete, testable, and scoped to produce artifacts you can inspect.

Project Anatomy

A project is a small filesystem contract. Once you know where things go, collaboration gets easier.

Fig 2. A typical IRL project layout: the plan file stays small and readable; raw inputs and derived data are separated; outputs are versioned artifacts; logs keep the loop honest. 

my-project/
├── main-plan.md    # your control surface
├── 02-data/
│   ├── raw/
│   └── derived/
├── 03-outputs/
└── 04-logs/

Rules That Make IRL Work

The “secret sauce” is boring engineering discipline applied to AI: scope, idempotency, and review.

Plan owns intent

If you want different behavior, edit the plan. Don’t rely on the assistant to remember “how we usually do it.”

Artifacts over prose

Prefer outputs you can inspect: HTML reports, CSVs, diagrams, scripts, tests. Make the work legible.

Idempotency by default

Rerun-safe steps. Use stable filenames. Detect and reuse prior outputs. Avoid “append forever.”

Diffs are the truth

Before you trust results, look at what changed. If it can’t be diffed, it’s hard to audit.

A small template for writing better plan steps 
## Task: [short name]

- Input(s): [files, links, assumptions]
- Output(s): [exact paths]
- Constraints: [idempotency rules]
- Done when: [verifiable checks]

CLI Commands (IRL Tooling)

IRL usually comes with a small CLI to scaffold projects and check your environment.

Command What it does
irl init "purpose" Create a new project scaffold (plan + folders).
irl templates List available templates.
irl update Refresh templates from the template repository.
irl doctor Check environment and tool availability.
