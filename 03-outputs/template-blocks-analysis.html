Proposed Block Library Structure

To implement a modular "Lego block" system, organize blocks into a library:

# blocks/
# ├── core/           # Universal (always include)
blocks/core/setup.md
blocks/core/pre-run-hooks.md
blocks/core/tasks.md
blocks/core/post-run-hooks.md

# ├── metadata/       # Pick one per template

Template Composition Example

A new template could be composed declaratively:

# template: data-science-manuscript
include:
  - core/*                    # All universal blocks
  - metadata/research         # Research project metadata
  - metadata/publication      # Journal submission info
  - strategy/academic-style   # APA 7, NIH formatting
  - strategy/data-science     # Reproducibility principles
  - docs/quarto-config        # Quarto website output
  - docs/references           # BibTeX management
  - quality/data-spec         # Data dictionary
  - quality/reproducibility   # Checksums, seeds
  - workflow/decision-log     # Audit trail

Key Insights

  • 4 blocks are truly universal (Setup, Pre-Run, Tasks, Post-Run) - these form the template skeleton
  • Academic Style block appears in 4 templates - high reuse value, should be standardized
  • Quarto Config block appears in 4 templates - could be auto-included when .qmd files are expected
  • Metadata blocks are mutually exclusive - pick one category per project type
  • Advanced workflow blocks (Decision Log, Maintenance) are powerful but underutilized - consider promoting
